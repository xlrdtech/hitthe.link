The Omni-inbox architecture debate

Mapped from qi's Second Me conversation (with the Higinio Ochoa persona). The question, every path proposed, why each was kept or rejected, and where the reasoning landed.
The goal
One inbox. Every channel. Live. His UI only.
All social DMs + SMS + email + AI web-chats piped into ONE stream in one app qi builds — read AND reply from it, full history, always-on, never open another app again.
Paths proposed → verdict
1
Matrix + bridges
Decentralized federation, bridges to nearly every protocol — the hacker default for control. qi already runs a mautrix stack.
Kept as option — wins only if decentralization/privacy is the priority
2
Self-hosted Mattermost / Rocket.Chat
Less bridge overhead, tighter API, plays clean with n8n webhooks. Lose federation, gain simplicity + faster to stabilize.
Viable if he doesn't care about protocol politics — speed over federation
3
API-first: custom n8n aggregation layer
Every platform with an API/webhook → routed into one stream. SMS via Twilio, email via catch-all parser, social DMs via notification webhooks. n8n normalizes to one payload.
REJECTED by qi's hard constraint — "I can't depend on the API being available at all." APIs get revoked/rate-limited/don't exist.
4
Transport-layer interception (past-API)
The pivot once API-first died. Capture BELOW the app: notification-scraping, DOM content-scripts, or wire-level.
This is the direction — immune to API deprecation
The winning architecture
A passive TAP, not a proxy
qi's decider: "MITM must catch everything everywhere 24/7 but must NEVER disconnect me or my remotes." A proxy sits in the path = a failure point. A tap is out-of-band — if it dies, real traffic is untouched.
  • Desktop: SPAN port / network TAP at the managed switch → mirror ALL traffic to a capture box running Suricata or Zeek in IDS mode (passive). Extract message payloads from WebSocket/XHR → forward to n8n. Invisible, always-on, can't break the connection.
  • Mobile: an always-on Android NotificationListener — pure observation, zero interference, catches even E2E messages after on-device decrypt → n8n webhook.
  • Desktop DMs / AI-chats: browser-extension content-script watches the DOM for new messages (web.whatsapp, messenger, IG, AI-chat UIs), snags content + injects into the input field for outbound. Brittle to UI changes but API-proof.
  • Email: the easy one — catch-all inbox + IMAP polling.
Interesting points worth keeping
API dependency is the thing that kills these setups. Building on platform goodwill = building on sand. Every resilient design here routes around the API entirely.
"Tap not proxy" is the whole insight. The constraint (never interfere with live traffic) forces out-of-band capture — which is also more resilient. The constraint and the best design point the same way.
Reverse the protocol once, own it forever. Wire-level interception is high upfront effort, but once mapped, no API shutdown can touch it.
Open question qi left: custom dashboard for read+reply, vs aggregate-for-visibility with replies still native. He wants read+reply in ONE UI — so: full custom frontend.
