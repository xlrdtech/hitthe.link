xen-conductor.js :8030Throttle

"Hermes's hands" — fans your voice/inject work across parallel persistent Claude lanes and narrates them through VVS. Same pattern as the Conductor.app you flagged.
Use case
Real value: instant parallel answers + a fast-responder lane + scope-routed work lanes, so one voice command can do many things at once without blocking. This is core to "never idle / parallel-is-the-default."
Importance
High — when you're driving it. Low when you're away: it keeps a fast lane, home lane, and 2 warm-pool lanes never reaped, all default-Opus except the haiku fast lane.
Why it burns
It spawned the three 35-hour lanes (45252/48840/50160). Work lanes reap after 20 min idle — but pool/fast/home never do. Idle warm-pool Opus lanes sit on your account indefinitely and process any inject that arrives.
BURN: 3 lanes × 35h · ~6 CPU-hrs total

xen-warm-lane.js :8021Throttle

A warm OpenAI-compatible backend for the Hermes agent — keeps one claude -p alive (lean config, no canon hooks) so Hermes replies in ~3-5s instead of cold-starting.
Use case
Makes "type XEN → talk to the Hermes agent" instant. Genuinely useful for the live agent front.
Importance
Medium. Only valuable while Hermes is actively serving requests.
Why it burns
It auto-respawns 2 seconds after any exit (built-in retry loop). If the underlying session hits the usage wall, it just respawns into the same wall — a low-grade perpetual retry against your account.
BURN: persistent + 2s respawn loop

XenOpenAIGateway :8020Keep (idle-safe)

Fork-mode OpenAI gateway: cold-starts a claude -p per API call (6-30s) so external tools can hit your Claude as an OpenAI endpoint.
Use case
BYOM bridge — lets anything that speaks OpenAI use your Claude. Real, but call-driven.
Importance
Conditional. Costs nothing when no calls arrive; spikes hard per call (heavy cold-start, flood-prone).
Why it burns
Only when called — but each call is an expensive full cold-start. Safe to leave armed; dangerous if something loops calls into it.
BURN: zero idle · spikes per call

Canon + auto-memory at SessionStartTrim

Not a process — the amplifier. Every session reloads CLAUDE.md cold-start (4 canon files) + MEMORY.md (374 lines / 30KB) ≈ 78KB before doing anything.
Use case
Crash-resume continuity — the whole point of the memory system. You never re-explain context.
Importance
High value, but over-weight. The continuity is essential; the size is the problem.
Why it burns
78KB on every relaunch pushes fresh sessions toward 95% before work starts → the documented compaction-loop that ate 25% of a session for nothing. It multiplies the cost of every lane above.
BURN: multiplier on all sessions
