Xen Architecture — JetStream as Source of Truth

Event-sourcing spec for all execution events. qi decree, 2026-07-22. The standard all live-data / state / execution work follows.

Core principle: NATS JetStream is the durable, append-only source of truth for every execution event. Browsers, agent sessions, and WebSocket connections are projections — never authoritative. Everything downstream is a fast read-model rebuilt from the log.

1Source of truth

  • JetStream = durable append-only log for ALL execution events; project into fast read-models. Never treat browser / agent session / WSS socket as source of truth.
  • Persists with configurable stream + consumer behavior; Nats-Msg-Id header for dedup.
  • No context lost when a cloud seat, phone, remote host, or browser restarts.
  • WSS clients reconnect with a stored JetStream cursor → receive missed events → resume live.

2Observability plane

Keep the existing OTEL collector + OpenObserve. Same trace IDs + event envelopes so one tool action follows across the whole chain:
WSS → NATS → browser adapter → terminal → worker
OTEL collectors support retries, queues, batching, and transformation/redaction before export.

3Command flow (CQRS)

  • Never run durable work directly on a WSS message.
  • A WSS command becomes a durable command-event first (persisted to JetStream), then executes.
  • The gateway returns accepted as soon as JetStream persists it — ack-on-persist, not ack-on-execute.
  • Publisher-side dedup via Nats-Msg-Id = your stable event ID. Per-subject discard policies for unbounded-dedup where apt.

4Universal event envelope + schema

  • Protobuf or JSON Schema for every type; codegen TypeScript + Python + Zig (schema = single source).
  • Payloads ≤ 256 KB. Versioning: backward-compat only, additive OK, breaking → V2.
  • Binary / large raw evidence → encrypted object storage + signed evidence-ref (the ref rides the message, not the blob).
  • Secrets NEVER in payloads — vault references + non-sensitive metadata only.

5Subject model

Do NOT put raw user strings, account names, URLs, or tokens into subject components. Subjects are routing keys, not a database / not state.

6State = event log + projection

  • Canon is a PROJECTION, never a hand-maintained competing source of truth.
  • Agents read a materialized Canon snapshot first, then subscribe to changes.
  • Keep verbatim first-source records chunked + indexed. Summaries are derived navigation aids, never replacements for raw terminal / browser / tool records.
  • Snapshot events every N events + on workflow terminal state → new seats load current state in milliseconds, then replay only the delta.

7Browser & HTTP discovery

  • Enroll a dedicated, owned browser profile into a loopback-only explicit interception proxy.
  • Record raw flows only to local encrypted evidence storage.
  • Keep normal personal browsing, SSH, Tailscale, remote-desktop OUT of it.
  • Publish only sanitized to JetStream: endpoint, method, schema-fingerprint, status, timing, evidence-ref. Never raw bodies / secrets on the bus.
Build status (honest): This is the TARGET architecture. Already in-stack: NATS JetStream MCP + NATS bridge, OTLP collector + OpenObserve. The live system today is SSE-projection-only — it does not yet have the durable JetStream log beneath it. Not-yet-built: the command-event gateway (ack-on-persist), the typed envelope schemas + codegen, JetStream-backed projections, snapshot/delta cold-start, and the owned-profile discovery proxy.
