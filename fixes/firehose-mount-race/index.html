// fix · 2026-05-23 · EX_CONFIG mount-race

Patch the 78-exit daemons.

Six launchd jobs share one root cause: /Volumes/tech_/ not mounted at RunAtLoad time. Two clean fixes — local-disk relocation OR WatchPath + lazy reload.
⚠ Symptom (current state)

Six daemons exit 78 (EX_CONFIG) on first launch. Survive on launchd's KeepAlive retry, sometimes succeed if the user has logged in and mounted the volume by then. Plist lints OK. Affected:

  • com.xen.beeper-firehose
  • com.xen.mail-firehose
  • com.exedus.xen-notif-firehose
  • com.xen.firehose-healthcheck
  • com.xen.beeper-live-index
  • com.xen.omnimind-heal (and likely sms-agent / xen.js if it reads from tech_)

Root cause

All plists reference /Volumes/tech_/qi_data/exedus/xen/<script>.py for ProgramArguments[1] + WorkingDirectory + log paths. At GUI-login when RunAtLoad=true fires, the external volume tech_ hasn't been mounted yet. Python script can't find its script-file or config — exits with EX_CONFIG (78). KeepAlive bounces it; depending on mount timing, it may catch on the second/third try, but the cycle leaves error logs and consumes restarts.

Fix A — WatchPath on mount point (least invasive)

✓ Add WatchPath + drop RunAtLoad

Tell launchd to fire when /Volumes/tech_/qi_data/exedus/xen/ appears, not at user-login. The job auto-fires once and only when the volume is actually mounted.

plutil -replace RunAtLoad -bool false ~/Library/LaunchAgents/com.xen.beeper-firehose.plist
plutil -insert WatchPaths -json '["/Volumes/tech_/qi_data/exedus/xen/beeper_firehose.py"]' ~/Library/LaunchAgents/com.xen.beeper-firehose.plist

# Reload
launchctl bootout gui/$(id -u)/com.xen.beeper-firehose 2>/dev/null
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.xen.beeper-firehose.plist
launchctl print gui/$(id -u)/com.xen.beeper-firehose | head -20

# Repeat for each affected plist

Fix B — Relocate to local disk (more permanent)

✓ Move scripts off /Volumes/tech_/

Copy the affected Python scripts + logs dir to a local-disk path (~/.xen/firehose-scripts/ or /Users/qi/firehose/) and update plist ProgramArguments[1] + WorkingDirectory + log paths accordingly. Eliminates the mount-race surface entirely. The tech_ volume can still hold data files (CSVs, large media), but executable scripts live on a always-mounted disk.

mkdir -p ~/.xen/firehose-scripts ~/.xen/firehose-logs
cp /Volumes/tech_/qi_data/exedus/xen/beeper_firehose.py ~/.xen/firehose-scripts/
cp /Volumes/tech_/qi_data/exedus/xen/mail_firehose.py ~/.xen/firehose-scripts/ 2>/dev/null
cp /Volumes/tech_/qi_data/exedus/xen/notif_firehose.py ~/.xen/firehose-scripts/ 2>/dev/null

# Patch plist paths
plutil -replace ProgramArguments.1 -string "$HOME/.xen/firehose-scripts/beeper_firehose.py" \
    ~/Library/LaunchAgents/com.xen.beeper-firehose.plist
plutil -replace WorkingDirectory -string "$HOME/.xen/firehose-scripts" \
    ~/Library/LaunchAgents/com.xen.beeper-firehose.plist
plutil -replace StandardOutPath -string "$HOME/.xen/firehose-logs/beeper-firehose.out.log" \
    ~/Library/LaunchAgents/com.xen.beeper-firehose.plist
plutil -replace StandardErrorPath -string "$HOME/.xen/firehose-logs/beeper-firehose.err.log" \
    ~/Library/LaunchAgents/com.xen.beeper-firehose.plist

# Reload
launchctl bootout gui/$(id -u)/com.xen.beeper-firehose 2>/dev/null
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.xen.beeper-firehose.plist

Recommendation

Start with Fix A (lower-risk, single-line plist edit). If it doesn't catch (e.g., if tech_ dismounts during operation), graduate to Fix B. Either path unblocks: SMS backlog ingestion · beeper firehose stream · mail firehose · notification firehose · firehose healthcheck.
