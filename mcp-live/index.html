MCP & Live Streaming Data

How an MCP server actually pushes real-time data — and the pattern to steal for Xen's live-status / voice-state dashboards. Sourced & cross-verified · spec 2025-11-25.

the crux Raw WebSocket is NOT a defined MCP transport. The spec defines exactly two — stdio and Streamable HTTP. Real server→client push is SSE layered inside Streamable HTTP, driven by resources/subscribe → notifications/resources/updated. Most "live data" MCP servers don't actually stream — they poll under a tools/call. That gap is the opening for Xen's build.

1The native push mechanism

Two standard transports only: stdio (local subprocess, newline-delimited JSON-RPC) and Streamable HTTP (remote). The MCP core team (Dec 2025) is explicit it'll "continue to support only two official transports." A WebSocket proposal (SEP-1287) exists but was not adopted — browsers can't set auth headers on WS, WS upgrades are GET-only (breaks POST-RPC), and SSE reuses standard HTTP infra. WS is only reachable via the Custom Transports escape hatch.

Push works over SSE inside Streamable HTTP: the client issues an HTTP GET to the single MCP endpoint with Accept: text/event-stream; the server holds it open and pushes server-initiated JSON-RPC notifications/requests down it — no polling. The spec-native real-time primitives:
  • resources/subscribe → server emits notifications/resources/updated on every change (needs the resources.subscribe capability). This is THE canonical push pattern.
  • notifications/resources/list_changed, notifications/progress, log notifications — all server-initiated notifications/*.

2Push vs. the polling shortcut

Confirmed: most "live data" MCP servers are not true streamers. They expose a tools/call (e.g. get_ticker) the agent re-invokes, or that hits a REST/WS API and returns one snapshot — request/response with polling underneath. The actual real-time mechanism is resources/subscribe + notifications/resources/updated over the SSE stream: server pushes on change, client never re-asks.
upstream feed (exchange WS / MQTT / IoT) → MCP server → on tick: notifications/resources/updated → client SSE stream → re-read

3The "Connected Domains" pattern verified + corrected

Real — but it lives in MCP-UI / MCP Apps, not core MCP resource _meta. When a resource carries an interactive UI (iframe app view — used by Goose / block), its metadata carries a CSP block _meta.ui.csp with:
  • connectDomains — fetch + WebSocket origins
  • resourceDomains — scripts / images / fonts
  • frameDomains — nested iframes
Adding an origin to connectDomains relaxes the iframe's same-origin/CSP restriction (the "SIM break") so the embedded UI may open a streaming connection — including ws:///wss:// — to another host. So the mechanics are right; the precise location is MCP-UI app-view _meta.ui.csp.connectDomains, defined by the MCP-UI/MCP-Apps spec Goose implements — not a field on a plain resource.

4Steal it for Zen Live Status / voice states

  • Ship Xen's live-status UI as an MCP-UI resource whose _meta.ui.csp.connectDomains whitelists your VVSVEI/XOS SSE-WS origin (e.g. wss://vvsvei.xlrd.org); the embedded dashboard then opens its own live socket straight to your existing SSE stream (canon 9) — the MCP host just renders it.
  • For agent-consumable status, also expose status://xen/live (and xen://live/voice-state) as a subscribable resource, pushing notifications/resources/updated on each state change — so any MCP client (Goose, Claude) gets Xen's live voice/call/agent status pushed, zero polling.
