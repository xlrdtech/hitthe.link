③ Do you need Hermes for memory? No.

Your use case
Instant · infinite · omnipresent memory

That's a storage + retrieval problem. Hermes is an LLM responder — a consumer of memory, not the memory itself. Routing recall through it adds latency + token burn to what should be a lookup.

voice inject → capture → store (bastra / sqlite / files) → recall on need → inject context. No LLM in that loop.
You already run the substrates
  • bastra — save_memory / load_memory / recall
  • sqlite — structured store
  • file memory + MEMORY.md index
  • NotebookLM — long-source corpora

Your own pain-map already called it: "WIRING not storage — 3 substrates exist; capture too late, recall only at SessionStart, no done-ledger." The gap is plumbing, not a missing engine.

When Hermes WOULD earn its tokens

Only as a consumer: compressing old context into memory entries, or answering "what do I know about X" in natural language over the store. Optional. Add when a summarizer measurably beats plain recall. Until then — waste of tokens, you're right.

④ The system you're actually building

Voice inject → auto-runs the SOP → you stay fully HOTL

The echo problem and the memory question are the same project: a multi-terminal session system where one voice command fans to the right lane, runs the SOP, and reports back — without you watching terminals.

The 3 wires that make it real:

  1. Router — voice inject → exactly one lane (kills echoes)
  2. Memory — capture-on-correction + recall-on-need + done-ledger (kills amnesia, no Hermes)
  3. Lane registry — named roles, dedup respawn (kills the stack of 6)

Build these three and the whole thing is HOTL: you speak, the right brain acts once, remembers it, and tells you — on a page like this, not a terminal.
