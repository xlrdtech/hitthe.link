Design Pattern Decision Tree

Pick a pattern by the friction you're feeling — not by memorizing names.
Design patterns rarely fail because they're wrong. They fail because we reach for them at the wrong moment. So don't start from the catalog — start from the pain. Name the friction you're actually hitting, and the pattern falls out of it.

①Where's the friction?

Three questions. Answer the one that's true right now — it routes you to the pattern.
Friction A · Creation
Is building the object the hard part — too many params, multi-step setup, or "which subclass?"
Builder→ many optional params / step-by-step assembly Factory→ caller shouldn't know which concrete class
Friction B · Boundaries
Is the pain at a seam — a third-party API, a legacy class, or an interface that doesn't fit yours?
Adapter→ wrap a mismatched interface to fit Facade→ hide a messy subsystem behind one door
Friction C · Behavior
Is behavior changing / branching — a growing if/switch, or "same call, different algorithm"?
Strategy→ swap one interchangeable algorithm at runtime State→ behavior depends on which mode the object is in

②The patterns, when to reach

Each has one job, a smell that signals it, and the smallest real example.

Builder

Reach when → an object needs many optional fields or multi-step assembly, and constructors are becoming telescoping messes.
Smell: new Thing(null, null, true, null, 3, false) — nobody remembers the arg order.
const query = new QueryBuilder()
  .from('users')
  .where('age', '>', 18)
  .orderBy('name')
  .limit(10)
  .build();  // readable, each step optional

Factory

Reach when → the caller shouldn't hard-code which concrete class to instantiate; the choice depends on input/config.
Smell: an if (type === 'x') return new X() chain copy-pasted in five places.
function makeParser(kind) {
  switch (kind) {
    case 'json': return new JsonParser();
    case 'csv' : return new CsvParser();
  }
}  // callers just ask for what they want

Adapter

Reach when → you have a class that does the right thing but the wrong interface (legacy code, a vendor SDK).
Smell: you're about to rewrite working third-party code just to change method names.
// your app expects .send(msg); the SDK gives .publish()
class NotifierAdapter {
  constructor(sdk){ this.sdk = sdk; }
  send(msg){ return this.sdk.publish({ body: msg }); }
}

Strategy

Reach when → one operation has several interchangeable algorithms and the if-ladder keeps growing.
Smell: if (mode==='fast'){...} else if (mode==='cheap'){...} else if... and it never stops.
const shipping = {
  fast:  o => o.weight * 2.0,
  cheap: o => o.weight * 0.8,
};
function cost(order, strategy){ return shipping[strategy](order); }
// add a strategy = add a key, touch nothing else

State

Reach when → an object behaves differently by mode, and transitions between modes are the real logic (draft → review → published).
Smell: every method starts with if (this.status === ...).
// each state owns its own transitions + allowed actions
const states = {
  draft:     { publish: () => 'review' },
  review:    { approve: () => 'published', reject: () => 'draft' },
  published: {} // terminal
};
