Mac Reference — bastra-recall Pattern

How memory works on M4
bastra-recall MCP runs as a daemon. Memories saved via save_memory() / load_memory(id) / recall(query).
Vault: /Volumes/X31/xen-vault/root/ on X31 PSSD.
Recall happens before any multi-step plan and at session start.
Save triggers: durable rules, finalized decisions, recurring frustrations, completed milestones.

Windows Setup — Once Candidate Chosen

Step 1 — Install and wire MCP
Clone / install chosen tool. Configure vault path (prefer dedicated SSD if available).
Add to %APPDATA%\Claude\claude_desktop_config.json MCP block.
Step 2 — Run as Windows service
Use nssm (Non-Sucking Service Manager) or sc.exe to install as auto-start service.
Restart policy: always. Log to C:\xen\logs\memory.log.
Step 3 — Sync vault to mesh
Mirror vault to Tailscale-accessible path so M4 can read Nitro memories and vice versa.
SyncThing or SMB share over Tailnet are canonical options (per OTLP/NATS mesh canon).
Step 4 — Verify cross-session recall
Save a test memory in Claude on Nitro. Kill + restart Claude. Recall the memory.
Only mark LIVE when qi confirms by voice or visual evidence — HTTP 200 is not verified.
