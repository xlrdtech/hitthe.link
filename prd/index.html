Xen Swarm-of-Swarms / Sintra Replacement

Fully replace Sintra with a self-hosted MCP + tmux + personas stack running on qi's own Claude Code seats. One brain on M4, one voice, a tailscale home-mesh as the nervous system, remote boxes as limbs. Free-first. Self-hosted. No SaaS in the loop.

Section 01

1Goals, Scope & Non-Goals

Retire Sintra entirely. Everything Sintra did — persistent AI helpers, an inbox, calendar, content generation, research, scheduling — is re-hosted as a swarm-of-swarms Xen operates on its own seats, with zero recurring SaaS spend.

Goals

  • Full functional replacement of Sintra — every "helper" capability qi used (drafting, research, scheduling, comms, image/video, memory of prior work) is served by Xen personas over MCP.
  • Self-hosted on qi's Claude Code seats — no per-seat SaaS billing; the reasoning runs on seats qi already owns, orchestrated locally.
  • Deterministic routing — a single spawn_agent path and one global config decide which persona handles which request; no manual wiring per task.
  • Durable memory across sessions — personas recall prior context via the core-knowledge module, so the swarm improves rather than restarting cold.
  • One brain, one voice — all personas speak through the singular Xen voice; personas differentiate by thread and prompt, never by a second narrator.

In Scope

Swarm topology

The three-tier Hypervisor → Cognitive → Execution structure, its data flow, and the mesh that connects M4 to remote limbs.

The five MCP modules

core-knowledge, integration-gateway, browser-ops, temporal-cron, media-forge — each as an addressable MCP server with a defined tool surface.

Agent orchestrator

The spawn_agent CLI, per-agent tmux sessions, persona prompt blocks, and the global ~/.claude.json config shape.

Config & routing scripts

The declarative persona registry, MCP server manifest, and the router that maps an inbound request to a persona + tool set.

Non-Goals

  • No paid SaaS dependencies — canon is free-first. Sintra, and any equivalent subscription helper platform, is removed, not re-subscribed. Paid rails are permitted only as a temporary bridge with a written exit plan.
  • No paid inference rails — reasoning runs on qi's existing seats; metered per-token API billing is out of scope for the default path.
  • No new hosted database or vector SaaS — memory is served by the existing self-hosted core-knowledge daemon, not a managed cloud store.
  • No second voice or second narrator — this PRD does not introduce any persona that speaks independently of Xen.
Canon tie-in: Free-first is non-negotiable. If any module here can only be met by a paid service, the module ships in a degraded free mode first and the paid path is filed as a dated bridge item — never the default.
Section 02

2Tiered Topology

Three tiers. The top tier decides, the middle tier plans, the bottom tier executes. Results flow back up the same spine. M4 is the brain; the tailscale mesh is the nervous system; remote boxes are limbs that act but never think for themselves.

Tier 1 · The Brain

Hypervisor

A single always-on Xen process on M4. Receives every inbound signal (voice, SMS, Beeper, schedule fire, browser event), classifies intent, and decides which cognitive persona owns it. Holds no long task logic itself — it is the router and the voice. It is the only tier that speaks.

decides & routes · owns the one voice · lives on M4 only
▼   dispatch (persona + task)   ▼
Tier 2 · The Mind

Cognitive Swarm

Persona-driven planners, one tmux session each (researcher, strategist, writer, comms, ops). A cognitive agent breaks a goal into concrete steps, chooses the MCP tools required, and emits execution jobs — it plans and reasons, it does not touch the outside world directly except through MCP.

plans & reasons · one persona per tmux session · reads/writes memory
▼   MCP tool calls (jobs)   ▼
Tier 3 · The Limbs

Execution Swarm

Stateless workers behind the five MCP modules. They browse, call APIs, render media, run cron fires, and read/write memory. They do exactly what the job says and return a structured result — no independent decisions. Runs on M4 and on remote boxes reached over the mesh.

executes only · stateless · returns structured results
▲   results bubble back up the spine   ▲

Data flow between tiers

  • Down (dispatch): Hypervisor classifies an inbound signal → selects a persona → hands the goal to that persona's tmux session as a prompt injection. Decision authority lives at Tier 1.
  • Middle (planning): The cognitive persona decomposes the goal, recalls relevant memory from core-knowledge, and issues one or more MCP tool calls. Planning authority lives at Tier 2.
  • Down again (execution): Each MCP call is a job for a Tier-3 worker. Workers are stateless and idempotent where possible; they never re-plan.
  • Up (results): Worker returns a structured payload → persona synthesizes → Hypervisor speaks the single-voice result and persists the outcome to memory. Nothing is dropped; every goal's thread persists.
Canon mapping: M4 is the brain (Hypervisor + memory live here, nothing load-bearing on boot disk). One brain, one voice (only Tier 1 narrates). Tailscale home-mesh is the nervous system (all inter-tier traffic rides the tailnet). Remote boxes are limbs (Tier-3 workers only — they never hold decision authority). Parallel-always: the cognitive and execution swarms fan out; sequential is blasphemy.

Section 03

3MCP Module Requirements

Five MCP servers form the entire tool surface of the swarm. Each is an independent, restartable process addressed over stdio or the tailnet. A persona sees only the tools it is granted in config. Every module below states its responsibilities, key tools/endpoints, and the acceptance bar it must clear to be considered shipped.

core-knowledge memory · recall

Responsibility: The swarm's durable memory. Persists facts, decisions, prior deliverables, and per-persona context so no session starts cold. Maps directly onto the existing bastra-recall daemon — this module is its MCP face, not a new store.

Key tools

  • recall(query, k) — semantic + keyword retrieval over the vault; returns ranked memory entries with source paths.
  • save_memory(text, tags, persona) — write a durable fact/decision; write-tmp-then-rename, never a sync write on a hot path.
  • load_memory(scope) — bulk hydrate a persona's working set at session start.
  • index_status() — health + entry count, so the Hypervisor can fail soft when memory flaps.

Acceptance: A fact saved by one persona is recalled verbatim by a different persona in a fresh session; memory calls fail soft (a flapping backend NEVER blocks voice/SMS); index_status returns entry count > 0 and p95 recall latency < 800 ms on M4.

integration-gateway external APIs · comms

Responsibility: One authenticated door to the outside world — email, SMS, Beeper/omni-inbox, calendar, and any REST connector a persona needs. Centralizes auth so no persona holds raw credentials and so rate-limits/retries are enforced in one place.

Key tools

  • send_message(channel, target, body) — unified send across SMS, Beeper, email; channel-agnostic to the caller.
  • read_inbox(channel, since) — triage-ready pull of recent messages with sender + thread identity.
  • calendar_op(action, event) — list / create / update / respond on the connected calendar.
  • rest_call(connector, method, path, body) — generic authenticated REST for whitelisted connectors; credentials injected from env server-side.

Acceptance: A persona sends and reads a message on ≥2 channels without ever seeing a token; a calendar event round-trips (create → list shows it); an unwhitelisted connector or a call carrying an inline secret is rejected; all creds resolve from env, never from the request.

browser-ops Playwright automation

Responsibility: Real-browser automation for anything without a clean API — logging into a site, scraping, filling forms, screenshotting for visual verification. Playwright is the chosen framework (Puppeteer was cut); runs M4-only against a persistent, logged-in browser context.

Key tools

  • navigate(url) / click(selector) / fill(selector, value) — core drive primitives.
  • extract(selector) — return page text/DOM for a persona to reason over.
  • screenshot(scope) — capture a PNG for visual verification (proves the user-facing surface, not just an HTTP 200).
  • session_keep(url) — hold a logged-in context open (e.g. keep a web app authenticated).

Acceptance: A scripted login to a real site succeeds and a follow-up action runs inside that authenticated session; a screenshot artifact is produced on demand and is used as the verification gate for a browser task; the browser context survives across separate tool calls (no re-login per call).

temporal-cron scheduling · wakeups

Responsibility: Time. Registers recurring jobs and one-shot wakeups, then fires them into the Hypervisor so the swarm acts on a clock without a human present. This is what replaces Sintra's scheduled/automated tasks.

Key tools

  • schedule(cron, persona, goal) — register a recurring job bound to a persona and a goal.
  • wakeup(at, persona, goal) — one-shot future fire.
  • list_jobs() / cancel(job_id) — inspect and retire schedules.
  • fire_now(job_id) — manual trigger for testing without waiting for the clock.

Acceptance: A job scheduled for T+60s fires within ±5s and lands as a real dispatch into the Hypervisor with the correct persona/goal; jobs survive a daemon restart (persisted, not in-memory only); list_jobs reflects registrations accurately; a cancelled job never fires again.

media-forge image · video · audio gen

Responsibility: Generative media — images, video, and audio/voice — so personas can produce visual and spoken output (hero images, clips, TTS lines) without a paid design SaaS. Wraps the existing free-first generation rails behind one MCP surface.

Key tools

  • gen_image(prompt, opts) — text-to-image / image-to-image; returns a saved asset path on M4.
  • gen_video(prompt|image, opts) — text/image-to-video clip generation.
  • gen_audio(text, voice) — TTS / voice synthesis for spoken deliverables.
  • list_assets(since) — enumerate produced media for reuse and to avoid re-rendering.

Acceptance: Each of image, video, and audio produces a real, openable artifact saved under the canonical M4 tree (never boot disk); a persona can request media and receive back a path it can attach to a comms send; the default path uses a free rail (paid generation only behind a dated bridge flag).

Section 04

4Agent Orchestrator Design

The orchestrator turns "route this goal to the strategist" into a running, MCP-equipped Claude Code seat in its own tmux session. One CLI, one config file, one persona-prompt convention. Everything is declarative so a new persona is a config entry, not a code change.

spawn_agent CLI

spawn_agent is the single entry point. Given a persona name and an initial goal, it: (1) reads the persona from the global config, (2) creates or reuses a dedicated tmux session, (3) launches a Claude Code seat there with exactly the MCP modules that persona is granted, and (4) injects the persona prompt followed by the goal. Idempotent — re-invoking for an existing persona re-attaches instead of spawning a duplicate.

# spawn the strategist persona and hand it a goal
spawn_agent --persona strategist \
            --session xen-strategist \
            --goal "Draft Q3 outreach plan for Self Exec, then schedule the send"

# under the hood it runs, per persona config:
tmux new-session -d -s xen-strategist \
  "claude --mcp-config ~/.claude.json \
          --allowed-mcp core-knowledge,integration-gateway,temporal-cron \
          --append-system-prompt \"$(cat personas/strategist.md)\""
tmux send-keys -t xen-strategist "$GOAL" Enter

tmux session model

  • One session per persona — named xen-<persona>, so state is inspectable and a stuck persona is restartable in isolation.
  • Long-lived — sessions persist between goals; memory + tmux scrollback preserve continuity.
  • Isolated tool grants — each session launches with only its granted MCP modules, enforcing least privilege at spawn time.
  • Attachable — qi (or the Hypervisor) can tmux attach to any persona to observe or intervene, never required to (HOTL by default).

Persona prompt block

Each persona is a markdown file under personas/. It is prepended as the system prompt at spawn. It defines the role, the granted tools, the output discipline, and the canon it must never violate. Personas differentiate by prompt and thread — the voice stays singular.

# personas/strategist.md
ROLE: Xen — Strategist persona. You plan; the execution swarm acts.
VOICE: You are Xen. Never announce yourself as a separate agent.
TOOLS: core-knowledge (recall/save), integration-gateway (comms,
       calendar), temporal-cron (schedule/wakeup).
METHOD:
  1. recall() relevant prior context before planning anything.
  2. Decompose the goal into concrete, parallelizable steps.
  3. Emit each external action as an MCP tool call — never act directly.
  4. save_memory() the decision + outcome when done.
DISCIPLINE:
  - Free-first. Never propose a paid SaaS as the default path.
  - Verify before claiming done; a screenshot/artifact beats an HTTP 200.
  - Nothing drops: every sub-goal persists until resolved.
  - Report status as verified / shipped-unverified / pending only.

Global config shape — ~/.claude.json

One file declares the MCP servers and the persona registry. The orchestrator reads it; nothing about routing is hardcoded.

{
  "mcpServers": {
    "core-knowledge":      { "command": "node", "args": ["mcp/core-knowledge.js"] },
    "integration-gateway": { "command": "node", "args": ["mcp/integration-gateway.js"], "env": { "CREDS_FROM": "env" } },
    "browser-ops":         { "command": "node", "args": ["mcp/browser-ops.js"] },
    "temporal-cron":       { "command": "node", "args": ["mcp/temporal-cron.js"] },
    "media-forge":         { "command": "node", "args": ["mcp/media-forge.js"] }
  },
  "personas": {
    "strategist": { "prompt": "personas/strategist.md", "mcp": ["core-knowledge","integration-gateway","temporal-cron"] },
    "researcher": { "prompt": "personas/researcher.md", "mcp": ["core-knowledge","browser-ops","media-forge"] },
    "writer":     { "prompt": "personas/writer.md",     "mcp": ["core-knowledge","media-forge"] },
    "comms":      { "prompt": "personas/comms.md",      "mcp": ["core-knowledge","integration-gateway"] },
    "ops":        { "prompt": "personas/ops.md",        "mcp": ["core-knowledge","integration-gateway","browser-ops","temporal-cron"] }
  },
  "router": { "default": "strategist", "rules": "routing/rules.json" }
}
Routing: routing/rules.json maps inbound intent → persona (e.g. "research / find" → researcher, "text / reply / draft" → comms, "schedule / remind" → ops). The Hypervisor consults it, falls back to the default persona, and calls spawn_agent. Adding a capability = a rule + a persona file. No orchestrator code change.
Section 05

5Security

The swarm touches real accounts and real money-adjacent surfaces. Three hard rules keep it safe and canon-clean: no plaintext credentials anywhere, no paid rails on the default path, and strict per-persona thread isolation.

No plaintext creds

Every secret resolves from the environment at server start; never from a request, a prompt, or a committed file. The integration-gateway is the only holder of live tokens, injected via env.

Rule: secrets live in env + .gitignore; a request carrying an inline secret is rejected; a repo scan finds zero plaintext tokens.

No paid rails

The default path uses free, self-hosted rails end to end. A paid service is allowed only as an explicit, dated bridge with an exit plan — never silently, never as the default.

Rule: no module's default path bills per use; any paid fallback is flagged and logged as a tracked bridge item.

Per-persona thread isolation

Each persona runs in its own tmux session with only its granted MCP modules. A persona cannot read another's session, and comms personas sign the correct thread — no cross-thread leakage.

Rule: least privilege at spawn; a persona granted N modules cannot call an (N+1)th; threads stay bound to their owning persona.

Enforcement checklist

  • Env-only secrets — .env + .gitignore cover every credential; CI/pre-commit scan blocks a commit containing a token pattern.
  • Least-privilege spawns — spawn_agent passes --allowed-mcp = exactly the persona's grant list; anything outside it is unreachable from that seat.
  • Gateway-only egress for creds — no persona calls an external API directly; all authenticated calls route through integration-gateway.
  • Thread binding — comms replies always sign the correct thread; a persona never posts to a thread it does not own.
  • Free-first audit — a periodic check confirms no default path silently fell back to a paid rail.

Sintra-parity capability bar (all must pass end-to-end)

  • Persistent helpers: A persona spawns, takes a multi-step goal, and completes it across more than one tool call without a human touching the loop.
  • Durable memory: Context saved in one session is recalled by a different persona in a later session — provably, by producing the recalled text.
  • Comms in & out: The swarm reads an inbound message and sends a reply on ≥2 channels through integration-gateway, with correct thread identity.
  • Calendar: An event is created and then listed back — a full round-trip through the gateway.
  • Research / browse: A persona logs into a real site, extracts content, and returns a screenshot as visual proof.
  • Scheduling: A job registered for the future fires on time into the Hypervisor and survives a daemon restart.
  • Media generation: Image, video, and audio each produce an openable artifact on the canonical M4 tree, attachable to a comms send.
  • Routing: Three different inbound intents route to three different personas automatically via routing/rules.json, with default fallback working.
  • One voice: All persona output surfaces through the single Xen voice — no second narrator appears anywhere.
  • Zero SaaS spend: The full parity run completes without invoking any paid subscription rail on a default path.
